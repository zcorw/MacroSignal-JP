# Stage 0. 平台与技术栈发现

## 已确认事实

- 项目目标是创建“日本宏观政策效果监控器”。
- 项目核心任务是自动收集、清洗、保存并分析日本宏观数据，生成 Markdown 报告和图表。
- 分析重点是区分日本政策带来的真实实际增长，还是通胀和名义 GDP 增长。
- 优先使用官方数据源，包括 ESRI、e-Stat、总务省统计局、财务省和日本银行。
- 若官方数据自动下载复杂，应支持 `data/raw/` 手动放置文件的方案。
- 运行结果应生成 `outputs/reports/japan_macro_report_YYYY-MM-DD.md` 和至少 5 类图表。
- 技术要求明确指定 Python 3.11+、pandas、requests、matplotlib、pyyaml、openpyxl、beautifulsoup4、pathlib、logging、类型提示。

## 推荐选择

- 交付形态：Python 数据工程项目 + 命令行运行工具 + Markdown 报告生成器。
- 核心用途：日本宏观政策效果监控、研究分析、周期性报告生成。
- 主技术栈：Python 3.11+，以 pandas 数据处理为核心，matplotlib 输出静态图表，YAML 管理配置。
- 前端 UI：Stage 0 暂不需要 Web 前端；报告和图表作为主要用户界面。
- 运行方式：本地脚本手动运行优先，同时保留未来接入定时任务或 CI 的空间。

## 暂定架构方向

- 使用 `src.fetch_data` 负责官方数据下载或手动文件入口校验。
- 使用 `src.clean_data` 负责原始数据标准化、字段命名、频率处理和保存。
- 使用 `src.indicators` 负责派生指标计算。
- 使用 `src.analyze` 负责评分和判断框架。
- 使用 `src.charts` 负责图表生成。
- 使用 `src.report` 负责 Markdown 报告生成。
- 使用 `run.py` 作为统一入口串联全流程。

## 决策理由

- 当前需求以数据采集、清洗、指标计算、报告生成和可复现研究为主，不需要复杂后端服务。
- Markdown 报告和静态图表足以支撑个人或小团队研究场景，也便于版本管理。
- 官方数据源下载稳定性不确定，因此必须把“自动下载”和“手动 raw 文件导入”都设计为一等路径。
- Python 生态更适合处理 Excel、CSV、时间序列和研究报告生成。

## 风险与缺口

- 官方数据源的真实下载接口、文件格式、编码、频率和历史版本需要后续验证。
- 是否需要定时任务、GitHub Actions 或云端部署尚未确认。
- 是否未来需要交互式仪表盘尚未确认；当前不纳入首版范围。
- USDJPY 可能不完全来自官方日本数据源，需后续确认是否允许使用 BOJ、FRED、Yahoo Finance 或手动文件。

## 后续修订

- Stage 4 后已确认需要轻量 Web UI。
- Stage 5 后已确认清洗后的结构化数据入库，Web UI 从数据库动态生成页面。
- 图表方案已从 `matplotlib` 预生成静态图，调整为浏览器端 ECharts 动态绘制。
- `src.charts` 不再作为首版核心模块，图表序列由数据库和 Web 层提供。
