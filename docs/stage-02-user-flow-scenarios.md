# Stage 2. 用户流程与运行场景

## 已确认事实

- 项目部署在 VPS 上。
- 系统需要定期执行命令，自动下载、清洗、分析并生成结果。
- 日本关键宏观数据多数有官方发布时间表，但发布时间、官网文件更新和页面同步可能存在延迟。
- 输出需要易于阅读理解，分析口径为保守研究分析。

## 已确认决策

- 定时策略：每日运行轻量检查，关键数据更新后生成正式报告。
- 默认运行时间：建议每天 `10:30 JST` 或 `11:00 JST` 执行一次，避开 GDP、CPI 等上午发布时间后的短暂延迟。
- 补跑机制：如果下载失败或关键数据当天应发布但未取到，当天晚间再跑一次。
- 正式报告触发条件：GDP、CPI、工资、JGB 或 USDJPY 中任一关键数据源出现新数据，生成新的日期报告。
- 输出形式：首版将结构化分析结果写入数据库，并通过轻量 Web UI 和 ECharts 动态展示。

## 推荐用户流程

1. VPS 上通过 cron 定时执行 `python run.py`。
2. 程序读取 `config.yaml` 中的数据源、运行模式和报告配置。
3. 下载模块检查官方数据源是否有新文件或新日期。
4. 若无新关键数据，则记录日志并可选择不生成正式报告。
5. 若发现新关键数据，则执行清洗、指标计算、评分、画图和报告生成。
6. 报告写入 `outputs/reports/japan_macro_report_YYYY-MM-DD.md`。
7. Web UI 从数据库读取最新报告、指标快照和图表序列，展示给用户。
8. 如果启用 Markdown 导出，`outputs/reports/latest.md` 指向或复制最新报告，便于文件级快速查看。

## 关键场景

### 场景 A：每日常规检查

- 触发：cron 每天上午执行。
- 预期：检查数据源、更新可用数据、记录运行日志。
- 若没有新数据：不重复生成正式报告，或仅更新运行日志。

### 场景 B：关键数据发布日

- 触发：GDP、CPI、工资等官方发布日。
- 预期：下载新数据，写入结构化分析结果，Web UI 动态生成完整报告和图表。
- 风险：官方页面可能晚于发布时间更新。
- 处理：失败或未发现新数据时，晚间补跑。

### 场景 C：市场数据日常更新

- 触发：工作日 JGB 或 USDJPY 更新。
- 预期：更新市场压力指标。
- 处理：若只有市场数据更新，可生成轻量报告或等到宏观数据更新后合并进正式报告。首版建议生成正式报告但在摘要中标注“宏观基本面数据未更新”。

### 场景 D：数据源不可用

- 触发：官网超时、文件结构变化、下载失败。
- 预期：不覆盖上一次成功报告。
- 处理：记录错误，保留失败日志，继续使用已有历史数据生成带缺口说明的报告或跳过报告生成。

### 场景 E：人工补充 USDJPY 长期数据

- 触发：BOJ 长期 USDJPY 自动化接口未确认。
- 预期：用户将手动 CSV 放入 `data/raw/manual/`。
- 处理：程序优先读取官方自动下载数据，缺失长期序列时读取手动文件，并在报告中标注来源。

## 已更新范围

- 需要 Web UI 快速查看分析报告和图表。
- 需要简短数据库保存运行记录、报告元数据和指标快照。

## 待确认问题

- 是否需要同时生成 HTML 版本报告。
- 是否需要运行失败通知，例如邮件、Telegram 或 Slack webhook。
- 历史报告保留策略是永久保留，还是只保留最近 N 份。
- 只有 JGB/USDJPY 这种市场数据更新时，是否每次都生成正式报告。
- Web UI 是否只允许自己访问，是否需要基础密码保护。
