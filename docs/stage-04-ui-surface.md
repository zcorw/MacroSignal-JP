# Stage 4. UI 表面发现与页面确认

## 已确认事实

- 项目需要轻量 Web UI。
- Web UI 用于快速查看分析报告和图表。
- 不需要账号管理。
- Web UI 可以公开访问。
- 首版使用单一样式，不需要深色模式或多主题。
- 用户是经济基础较少的读者，因此界面应结论清晰、解释充分、避免信息过载。

## 已确认决策

- Web UI 定位：公开只读报告查看界面。
- 首版不做登录、注册、用户权限、后台管理。
- 单一视觉风格：浅色、报表型、信息密度适中。
- 技术建议：Python Web 框架 + 服务端模板渲染，避免复杂前端工程。
- 页面内容以“快速理解当前判断”为优先，不做复杂交互式仪表盘。

## 页面清单

### 1. 首页 / 最新报告

路径建议：`/`

目标：让用户在 30 秒内知道当前日本宏观政策效果更偏向哪种解释。

页面元素：

- 最新报告标题和生成时间。
- 总结判断标签，例如：
  - 真实增长证据增强
  - 名义增长成分偏高
  - 居民购买力仍承压
  - 通胀压力上升
  - 财政压力需警惕
  - 数据不足，需继续观察
- 3-5 句总结判断。
- 三个核心评分：
  - 真实增长评分。
  - 通胀压力评分。
  - 财政压力评分。
- 关键证据列表，每条证据必须能对应到具体指标。
- 最新核心仪表盘表格。
- 最新核心图表预览：
  - nominal_gdp_yoy vs real_gdp_yoy
  - real_wage_yoy vs CPI_yoy
  - private_consumption_yoy
  - private_investment_yoy
  - USDJPY and JGB 10Y yield
- 进入完整报告详情的链接。

### 2. 历史报告

路径建议：`/reports`

目标：按时间追踪判断变化。

页面元素：

- 报告列表，按生成日期倒序。
- 每份报告显示：
  - 报告日期。
  - 总结判断标签。
  - 三个评分。
  - 数据覆盖日期。
  - 是否存在缺失数据。
- 报告详情入口。

### 3. 报告详情

路径建议：`/reports/{report_id}`

目标：阅读完整分析和全部图表。

页面元素：

- 完整 Markdown 报告渲染结果。
- 报告关联图表，使用 ECharts 动态绘制。
- 数据源状态摘要。
- 手动数据使用说明，如果有。
- 下载 Markdown 文件的链接。

### 4. 数据源状态

路径建议：`/sources`

目标：判断系统是否正常运行，哪些数据源已更新或失败。

页面元素：

- 最近一次运行时间。
- 最近一次运行状态：成功、部分成功、失败。
- 各数据源状态：
  - ESRI GDP
  - e-Stat GDP
  - e-Stat CPI
  - e-Stat Monthly Labour Survey
  - MOF JGB
  - BOJ / USDJPY
- 每个数据源显示：
  - 最新数据日期。
  - 最近下载时间。
  - 状态。
  - 失败原因或缺口说明。
- 最近几次运行日志摘要。

## 设计原则

- 首页结论先行，不要求用户先读完整报告。
- 图表不堆叠过多，首屏优先展示评分和判断。
- 表格字段用中文解释，避免只显示英文指标名。
- 每个专业术语在报告详情中提供短解释。
- 公开页面不展示敏感服务器路径、密钥、环境变量或详细异常堆栈。
- 数据源失败信息对外展示为可读摘要，完整堆栈只写入服务器日志。

## 静态 HTML 原型

按技能规则，Stage 4 确认页面后需要生成静态 HTML 原型到 `ui-static/`。原型只作为设计和结构确认，不连接真实数据。

## 后续架构影响

- 需要数据库表保存报告索引、指标快照、数据源状态和运行记录。
- Web UI 从数据库读取清洗后数据、指标、评分和报告段落，并用 ECharts 动态绘制图表。
- Markdown 报告只作为可选导出或归档。
- 公开访问但无账号管理，因此部署时应避免暴露管理接口和敏感文件。
